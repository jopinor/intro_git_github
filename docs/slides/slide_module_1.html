The version control problem

A general example from manuscript revision:

How would you name these files?

  • manuscript_FA.docx
  • manuscript_FA_LA.docx
  • manuscript_FA_LA_FA.docx
  • manuscript_final.docx

Or by date?

  • manuscript_20221001.docx
  • manuscript_20221224.docx
  • manuscript_final.docx

Warning

Both approaches break down: which file had the figure you liked? Which comment thread belongs to which version? Renaming files is not version control.

What is Git?

This is exactly the problem Git was built to solve.

  • A distributed version control system (VCS)
  • Tracks changes to files over time, with a message explaining why
  • Works locally — no internet needed
  • Free and open source

Git ≠ GitHub

These are often confused, but they’re different tools:

  • Git = the version control tool running on your machine
  • GitHub = a website that hosts Git repositories online

We’ll use both today — Git is what you use, GitHub is where you share.

Version control — the better way

One file. Many snapshots, each with a message explaining why it changed.

The filename never changes. The history does.

“Why not just use Google Docs?”

A fair question — for a single shared document, Google Docs works fine.

Git earns its complexity when you have:

  • Code and scripts, not prose — line-by-line diffs that mean something
  • Parallel experiments — try a new analysis approach without breaking the working one (branches)
  • Exact reproducibility — re-run the exact code that produced Figure 2, six months later

Why Git shines

The final advantage:

  • No internet required — full history available offline, on your machine

Every other tool requires a server connection. Git works everywhere — on planes, in the field, during internet outages.

Git vs Google Docs: Rule of thumb

Note

Collaborating on prose? → Google Docs (faster, simpler)

Collaborating on code/analysis? → Git (reproducible, historical, safe experiments)

Types of Version Control

Git is one kind of version control system — here’s how it compares to the alternatives.

Local VCS

No collaboration possible — if your laptop dies, so does the history.

Central VCS (e.g. SVN)

One single point of failure. No internet → no work.

Distributed VCS (Git)

Every clone is a full backup. Works offline. This is Git.

Git is winning

Git overtook SVN because every clone is a full backup — the distributed model won.

Why Git won

With Git’s distributed model:

  • Every copy is a full backup — no single point of failure
  • Everyone can work offline
  • Git is fast (no server communication needed)
  • Easy to merge work from many contributors

GitHub: Building on Git

GitHub built a free hosting platform on top of Git, adding:

  • Project management (Issues, Projects)
  • Automated testing (Actions)
  • Easy collaboration across teams
  • A social network for developers

The 6 core concepts — part 1

Git has hundreds of commands. You only need 6 ideas to be effective:

  1. Repository — a folder Git is watching
  2. Working directory — your files, right now, as you edit them
  3. Staging area — a holding pen for what goes into the next snapshot

These are the “places” where your code lives during development.

The 6 core concepts — part 2

The rest of the 6:

  1. Commit — a saved snapshot with a message
  2. Branch — an independent line of commits
  3. Remote — a copy of the repository living somewhere else (GitHub)

These are the “actions” you take with those places.

Your navigation tool

Tip

Everything from here on is one of these 6 things, or a relationship between two of them. If you ever feel lost, ask: “which of the 6 actions am I touching right now?”

GitHub Terminologies — Core

Now that you have the 6 concepts, here’s the vocabulary GitHub adds:

Term What it means
Repository A folder tracked by Git
Commit A saved snapshot with a message
Branch A separate line of development

GitHub Terminologies — Collaboration

Tools for working with others:

Term What it means
Fork Your own server-side copy of someone else’s repo
Pull Request A proposal to merge your changes into the main repo
Issue A task, bug report, or discussion thread
Organization A shared account for teams

Writing on GitHub: Markdown

Before we open GitHub for real, one last tool: GitHub renders .md files as formatted text automatically. Structure first:

# Heading 1
## Heading 2

**bold**   *italic*   `inline code`

- list item 1
- list item 2

Markdown: tables & links

| Column 1 | Column 2 |
|----------|----------|
| row 1 a  | row 1 b  |

[link text](https://github.com)

You’ll use exactly this much Markdown in the next two exercises — nothing more.
