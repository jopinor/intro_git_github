Module 3: Branches & parallel work

You’ve mastered the basic workflow: init → add → commit → log.

Now the real power: try new ideas without breaking what works. That’s what branches do.

Why branches?

Branches let you work on something new without breaking what already works.

Why branches? (continued)

In research: use a branch to try a new analysis approach. If it works → merge. If not → delete without any harm to main.

Tip

Discussion: what’s something you’ve wanted to try in your own analysis, but were afraid to risk breaking what already works?

5 reasons to branch

  1. Experiment — try a different method without risking what works
  2. Feature — build something new in isolation (e.g. a new plotting function)
  3. Bug fix — fix something urgent without mixing in unrelated work-in-progress
  4. Collaboration — you and a labmate work on the same project without colliding
  5. Review — let a colleague review your changes before they reach main

Why branches? (summary)

Different reasons, same mechanism: branches are cheap, disposable, and safe.

Use them liberally. Don’t be afraid to create one, experiment, and delete it if it doesn’t work out.

git branch + git checkout

Time to do what the diagram a moment ago showed, for real:

# See which branch you are on
git branch

# Create a new branch for a new normalization approach
git branch try-new-normalization

# Switch to it
git checkout try-new-normalization

# Create AND switch in one step
git checkout -b try-new-normalization

Making commits on a branch

You’re now on the new branch. Commits here behave exactly like before — they just don’t exist anywhere else yet:

# You are on branch "try-new-normalization"
nano analysis_log.md    # document the alternative approach

git add analysis_log.md
git commit -m "tried quantile normalization instead of log2"

# Go back to main — this note does not exist here
git checkout main
cat analysis_log.md

Branches are isolated. Changes on a branch don’t appear on main until you merge.

git merge

# On main branch, once you decide the new approach is good:
git merge try-new-normalization

Git replays the commits from the branch onto main. After merging, you can delete the branch:

git branch -d try-new-normalization

That merge went smoothly because nothing else on main had touched the same file. Merges aren’t always this clean.

Merge conflicts

Conflicts happen when two branches change the same line differently — e.g. you and a labmate both edit the conclusion in results_summary.md.

<<<<<<< HEAD
Treated samples show a 2-fold increase in expression.
=======
Treated samples show no significant change in expression.
>>>>>>> try-new-normalization

Resolving merge conflicts

When you git merge and Git can’t automatically resolve, it stops and marks the conflicts.

How to resolve:

  1. Open the file with conflict markers
  2. Choose which version to keep (or combine them)
  3. Remove all <<<<<<<, =======, >>>>>>> markers
  4. git add <file> → git commit

Done! The merge completes.

Exploring git history

# Full history
git log

# Compact — one line per commit
git log --oneline

# Go to a specific commit (read-only — "detached HEAD")
git checkout a499ea4

# Return to the latest
git checkout main

Ignoring files with .gitignore

Create a file called .gitignore in your repo root:

# Ignore raw instrument output (too large/sensitive for GitHub)
raw_data/

# Ignore temporary analysis files
*.tmp

# Ignore a specific file
secrets.txt
git add .gitignore
git commit -m "added gitignore"

Git will never ask you about these files again.

git reset — undoing things

Command Effect
git reset <file> Unstage a file (keep changes)
git reset --soft HEAD~1 Undo last commit, keep changes staged
git reset --hard HEAD~1 Undo last commit, discard all changes

Warning

--hard is permanent. Use with care.
