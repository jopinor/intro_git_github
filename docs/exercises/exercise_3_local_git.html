Exercise 3 — Local Git Workflow

Module 2 | ~35 minutes

Goal

Build a small, realistic project — experiment-tracker — and practice the core local Git workflow: init → add → commit → log → diff.

This is the running example we will keep extending for the rest of the day.

As you work: Practice the one-unit rule for commits and use clear, imperative commit messages from the slides (e.g., “add QC step” not “stuff done”).

Part A — Create the repository

cd ~/Downloads

mkdir experiment-tracker
cd experiment-tracker

git init
ls -a     # confirm .git was created

Part B — First commit: sample metadata

nano samples.csv
sample_id,condition,replicate
S01,control,1
S02,control,2
S03,treated,1
S04,treated,2

Save and close: Ctrl+X → Y → Enter

git status              # git sees the file but isn't tracking it
git add samples.csv
git status              # now it's staged
git commit -m "added sample metadata for 4 samples"
git log

Question: After git add samples.csv, you run git status again. Before checking the answer below — what do you think the wording changes to?

TipClick to reveal the answer 
On branch main

No commits yet

Changes to be committed:
  (use "git rm --cached <file>..." to unstage)
        new file:   samples.csv

“Untracked” became “Changes to be committed” — the file moved from the working directory into the staging area.

Part C — More commits: documenting analysis steps

nano analysis_log.md
# Analysis Log

## Step 1 — Quality control
Checked all 4 samples for read quality. All passed QC thresholds.
git add analysis_log.md
git commit -m "added analysis log with QC step"

nano analysis_log.md

Append:

## Step 2 — Normalization
Applied log2 normalization to raw counts before comparison.
git add analysis_log.md
git commit -m "documented normalization step"

git log --oneline    # see all 3 commits

Part D — Inspect changes with diff

nano analysis_log.md   # add a "Step 3 — Differential expression" section

Before staging:

git diff               # working directory vs staging area

After staging:

git add analysis_log.md
git diff --staged      # staging area vs last commit
git commit -m "documented differential expression step"

Compare your first and last commit:

git log --oneline             # note the two hashes
git diff FIRST_HASH LAST_HASH

Check your understanding

Question: You ran git diff before staging and saw nothing printed, even though you definitely edited analysis_log.md. What did you forget?

TipClick to reveal the answer

You probably forgot to save the file in nano before running git diff — Git can only see what’s actually written to disk. (Or, less likely, you already ran git add for that change, which would move it from git diff into git diff --staged instead.)

Question: What is the difference between git diff and git diff --staged?

TipClick to reveal the answer
  • git diff → compares your working directory to the staging area (what you’ve changed but not yet staged).
  • git diff --staged → compares the staging area to the last commit (what will actually go into your next commit).

If you run git add on a file, it “moves” out of the first diff and into the second.

Gotcha: empty folders won’t show up

If you create an empty folder and commit, Git will ignore it. This sometimes surprises people. If you need an empty folder (e.g., for output files), create an empty .gitkeep file inside:

mkdir output
touch output/.gitkeep
git add output/.gitkeep
git commit -m "add output directory"

This is a harmless trick — .gitkeep has no special meaning to Git. It’s just a placeholder file.

Command What it does
git init Create a repository
git status See what has changed
git add <file> Stage a file
git commit -m "msg" Save a snapshot
git log --oneline View history
git diff See unstaged changes
git diff --staged See staged changes
